Single Responsability Principle ( S )
Acest principiu se refera la faptul ca fiecare clasa trebuie sa aiba o singura 
responsabilitate, in caz contrat se la duce la iutilitatea ei si rescrierea codului.
Open-close Principle ( O )
Clasele ar trebuie sa fie deschise pentru extensii dar totusi inchise pentru 
modificari.
Liskov substitution Principle ( L )
Obiectele pot fi inlocuite mereu cu instante ale claselor derivate fara ca acest lucru
sa afecteze functionalitatea aplicatiei.
Interface Segregation Principle ( I )
Mai multe interfere sunt oricand de preferat in locul unei singure interfete generale,
astfel nu riscam ca prin modificarea "contactului" unui client sa modificam si 
contractele altor clienti.
Dependency inversion Principle ( D )
O clasa trebuie sa depinda de abstractizari, niciodata de obiecte concrete.
